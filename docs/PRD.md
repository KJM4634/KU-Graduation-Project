# PRD: 재난 잔해지대 인명탐지 및 구조지원 시스템 (DSAR-Assist)

## 0. 문서 정보
- 프로젝트명(가안): DSAR-Assist (Disaster Search And Rescue Assist)
- 재작성 배경: 개발 환경 이전(맥북 → Windows/RTX 3090)으로 인한 재구축. 이전 환경의 진행 상황(스택 결정, 데이터 전략, 초기 모델 실험 경향)은 본 PRD에 반영되어 있으며, 실제 파일(코드/데이터/가중치)은 복구 불가하여 재생성 필요. (최초 작성 시 GPU를 RTX 5080으로 가정했으나, Phase 1에서 실제 장착 GPU가 RTX 3090임을 확인하여 전체 문서를 정정)
- 개발 환경: Windows, GPU RTX 3090 (Ampere, sm_86) (Jetson Orin Nano는 엣지 배포 타겟이며, 개발/학습은 이 PC에서 진행)
- 전제: 개발 기간 제약 없음(완성도 최우선), 팀 4인 역할 분담 확정

---

## 1. 배경 및 문제 정의

### 1.1 문제 상황
대규모 지진(진도 9급 등)으로 건물이 완전 붕괴되어 잔해가 **야외 평지형**으로 형성된 재난 현장(건물 내부 잔존 공간이 아니라, 구조물이 개활지 수준으로 붕괴되어 평평해진 지형), 또는 산악 조난 현장에서:
- 119 구조대원은 시야가 제한적이고, 잔해에 매몰되거나 기절/의식불명 상태로 소리를 낼 수 없는 요구조자를 놓치기 쉽다.
- 구조대원 1인이 커버하는 구역과 실제 탐지된 요구조자 위치가 관제센터(현장지휘소)에 실시간으로 공유되지 않아, 구조 인력 배치가 비효율적이다.

### 1.2 기존 프로젝트와의 차별점
- 기존(지게차 AI관제)은 "충돌 예방"이 목적. 본 프로젝트는 "탐지 사각지대 보완 + 인력 배치 지원"이 목적으로 문제 본질이 다름.
- 핵심 기술 차별점: 일반 사람 탐지가 아닌 **잔해에 부분적으로 매몰/가려진 사람 탐지**에 특화. (COCO 등 일반 데이터셋 기반 모델의 한계 지점을 정면으로 다룸)
- 시나리오를 "완전 붕괴된 평지형 잔해지대"로 한정하여, 지진과 산악구조 두 시나리오가 "GPS가 살아있는 야외 불규칙 지형에서의 매몰 인명 탐지"라는 동일한 기술 문제로 통합됨 → SLAM 없이 GPS+YOLO 좌표 태깅만으로 시스템 완결 가능.

### 1.3 프로젝트 목표
1. 안전모 부착형 웨어러블 디바이스로 구조대원의 시야 사각지대를 YOLO 기반으로 실시간 탐지
2. 탐지된 요구조자 위치를 GPS 좌표와 결합해 서버로 전송, 지도 위 마커로 시각화
3. 구조대원 개인 LCD에 실시간 알림 제공 (놓친 요구조자 존재 여부)
4. 관제센터(현장지휘소) 대시보드에서 전체 구조대원 위치/이동경로/탐지 이력을 통합 관제
5. (확장) 구조대원별 커버 구역과 탐지 밀도를 기반으로 인력 재배치 제안

## 2. 스코프 정의

### 2.1 In Scope (반드시 구현)
- 잔해 매몰/부분 가림 인체 탐지 모델 (YOLO11 파인튜닝)
- 4방향 카메라 + Jetson Orin Nano 기반 엣지 추론
- GPS 모듈 기반 위치 태깅
- 탐지 시 부저 알림 + LCD 알림 표시
- 서버 전송: 탐지 이벤트(좌표, 시각, 스냅샷, 구조대원 ID)
- 관제센터 웹 대시보드: 지도 위 실시간 마커 시각화, 구조대원별 이동경로, 탐지 이력 확인
- 구조대원 개인 뷰: 본인 담당 구역 지도, 본인이 놓친(탐지된) 위치 확인

### 2.2 Out of Scope (이번 단계에서 제외)
- 실시간 SLAM 기반 3D 지도 생성 (GPS 태깅 방식으로 대체)
- 건물 내부 등 GPS 완전 음영구역 대응 (개활지 잔해지대로 시나리오 한정)
- 다중 디바이스 간 메시 네트워킹 (서버 중심 단일 업링크로 단순화)
- 인력 배치의 자동 재배치 실행 (1차는 "제안" 수준까지)

### 2.3 확장 과제 (여유 있을 시)
- 구조대원 커버리지 영역 계산(GPS 이동경로 기반 폴리곤 추정) 및 미탐색 구역 시각화
- 다수 구조대원 탐지 결과 클러스터링으로 중복 탐지 제거
- 배터리/네트워크 끊김 대비 오프라인 큐잉 후 재전송 (기본 예외처리는 1차 범위에 포함)

## 4. 하드웨어 구성

| 부품 | 수량 | 역할 |
|---|---|---|
| Jetson Orin Nano | 1 | 엣지 추론(YOLO), 센서 데이터 통합, 서버 통신 (배포 타겟) |
| 카메라 모듈 | 4 | 안전모 전/후/좌/우 시야 확보 |
| 7인치 터치 LCD | 1 | 구조대원용 실시간 알림/미니맵 표시 |
| GPS 모듈 | 1 | 구조대원 및 탐지 지점 위치 태깅 |
| 부저 모듈 | 1 | 탐지 즉시 청각 알림 |

### 4.1 개발/학습 환경과 배포 환경의 분리
- **모델 학습**: Windows PC (RTX 3090, Ampere, sm_86)에서 진행 — CUDA 가속으로 대규모 데이터 학습 가능
- **엣지 배포**: Jetson Orin Nano — 학습된 모델을 ONNX → TensorRT(FP16)로 변환 후 배포
- Jetson 실물 접근이 어려운 기간에는 RTX 3090 PC에서 학습 및 로컬 추론 속도까지만 측정하고, Jetson 실측(FPS, 발열, 전력)은 하드웨어 확보 시점에 별도 체크포인트로 재실행

### 4.2 하드웨어 배치 고려사항
- 안전모에 카메라 4개 + Jetson 부착 시 무게중심, 발열, 배선 정리 방안 별도 설계 필요
- LCD는 헬멧 부착이 시야를 가릴 수 있어, 손목형 거치대 또는 가슴 스트랩 거치 방식도 검토
- 흔들림에 따른 모션 블러 대응: 프레임레이트/노출시간 실험적으로 튜닝

## 5. 시스템 아키텍처

```
[안전모 유닛: Jetson Orin Nano]
  ├─ 카메라 x4 → YOLO11 추론(잔해 매몰 인체 탐지 파인튜닝 모델, TensorRT)
  ├─ GPS 모듈 → 현재 위치 스트리밍 (2초 주기)
  ├─ 탐지 발생 시 → 부저 알림(confidence≥0.5) + LCD 알림(confidence≥0.25)
  └─ 탐지 이벤트(좌표+시각+스냅샷+구조대원ID) → 서버 전송 (Wi-Fi/REST)
        │
        ▼
[서버: FastAPI + PostgreSQL/PostGIS]
  ├─ 탐지 이벤트 저장
  ├─ 구조대원 위치 실시간 수신/저장
  └─ 대시보드에는 WebSocket으로 실시간 반영, 로컬 SQLite로 오프라인 큐잉 대응
        │
        ▼
[관제센터 웹 대시보드: React + TypeScript + Kakao Map]
  ├─ 지도 위 구조대원 실시간 위치 마커 + 이동경로
  ├─ 지도 위 탐지된 요구조자 마커 (미확인/확인됨 색상 구분, 클릭 시 스냅샷)
  └─ (확장) 커버리지 영역 시각화, 인력 배치 제안
```

## 6. 확정된 기술 스택 (이전 환경에서 결정, 유지)

| 항목 | 결정 | 핵심 이유 |
|---|---|---|
| 탐지 모델 | YOLO11 → ONNX → TensorRT(FP16) | Jetson Orin Nano 공식 검증 조합, v8 자료 재사용 가능하며 성능 개선 |
| 4채널 카메라 처리 | 독립 추론(스티칭 없음) | 겹치는 시야 없어 스티칭 실익 없음, 카메라=탐지방향 자연 매핑 |
| 데이터 전략 | 공개데이터(occlusion 라벨 포함) + 합성 occlusion augmentation(주력) + 자체 촬영(검증) | 규모와 현실성 상호 보완 |
| GPS | UART + pyserial/pynmea2, 2초 주기 | 배터리/네트워크 부담과 경로 매끄러움 절충 |
| 부저/LCD | GPIO 부저, LCD는 PyQt 경량 네이티브 UI | 추론 파이프라인과 자원 경합 최소화 |
| 서버 | FastAPI + PostgreSQL/PostGIS | 엣지와 언어 통일, PostGIS로 커버리지 폴리곤 확장 대비 |
| 대시보드 | React + TypeScript + Kakao Map | 확정 |
| 통신 | Wi-Fi + HTTP REST, 서버→대시보드 WebSocket, 로컬 SQLite 오프라인 큐 | 데모 환경 단순화 |
| 신뢰도 임계값(잠정) | 로그/미니맵 ≥0.25, 부저 ≥0.5 | 미탐 비용 > 오탐 비용 우선, 실측 후 재조정 예정 |
| **라벨링 방식(신규 확정)** | **전체 범위(full-extent) bbox** — 가려짐과 무관하게 사람의 실제 전체 범위를 라벨링 | 초기 스모크테스트(61장, YOLO11n)에서 visible-only 대비 recall(0.748 vs 0.297), mAP50(0.812 vs 0.304) 모두 확연히 우세. 표본이 작아 본 학습에서 재검증 필요하나 방향성은 뚜렷함 |
| 학습 환경 | Windows + RTX 3090 (Ampere, sm_86, CUDA) | 개발 환경 이전으로 확정, Jetson은 배포 전용 타겟으로 분리. PRD 최초 작성 시 RTX 5080으로 가정했으나 Phase 1에서 실제 장착 GPU가 RTX 3090임을 확인하여 정정 (상세: docs/STACK.md) |

## 7. 데이터셋 구축 전략

### 7.1 공개 데이터셋
| 데이터셋 | 우선순위 | 비고 |
|---|---|---|
| COCO (person) | 상 (기반) | YOLO11 사전학습 가중치의 기반, CC BY 4.0 |
| CrowdHuman | 상 (occlusion 학습 핵심) | vbox/fbox 분리 제공, 비상업 연구용 한정·재배포 금지 |
| WiderPerson | 상 (부분가림 개념 최유사) | "partially-visible person" 전용 클래스 제공 |
| OCHuman | 중~상 (극한 occlusion 검증) | 사람 간 극한 상호 가림 벤치마크 |
| SARD/Heridal/C2A 등 드론·항공 시점 재난 데이터셋 | 하 | 헬멧캠(지상 시점)과 화각 불일치로 우선순위 낮춤 |

### 7.2 합성 Occlusion Augmentation
- 절차적으로 생성한 잔해 텍스처(콘크리트/철근/먼지/천막)를 bbox 영역에 합성
- occlusion 비율 30/50/70% 단계별 제어 (실측 비율 ±5% 허용오차), metadata.csv로 기록
- 학습셋 기준 합성 데이터 비율 **50~60% 상한 권장** (질감 편향 위험) — 실제 촬영 잔해 텍스처로 대체 시 60~70%까지 완화 가능

### 7.3 자체 촬영 (검증용 핵심 데이터)
- **실제 사람**(마네킹 아님)을 대상으로 촬영 — 팀원 C, D가 진행
- 안전 원칙(필수 준수):
  - 실제 잔해(콘크리트/철근 등 무거운 자재)를 사람 위에 올리지 않음
  - 스티로폼, 종이박스, 천, 우레탄폼 등 가벼운 소품으로 잔해처럼 보이게 구성
  - 촬영 대상자가 불편하거나 위험을 느끼면 즉시 중단
  - 장시간 같은 자세 유지 방지를 위한 세션 간 휴식 포함
- 목표 500~1,000장, occlusion 비율(30/50/70%)별 소품 배치, 각도/거리/조명 다양성 확보
- 라벨링 원칙: 전체범위(full-extent) bbox로 통일 (6장 참고)

### 7.4 분할 비율
| 소스 | train | val | test |
|---|---|---|---|
| 공개+occlusion 서브셋 | 80% | 10% | 10% |
| 합성(synthetic) | 80% | 10% | 10% |
| 자체 촬영(실사람) | 30% | 20% | **50%** (실사용 성능 증거이므로 test 비중 높게) |

## 8. 초기 모델 실험 결과 (이전 환경, 재검증 필요)

이전 개발 환경(맥북)에서 진행한 스모크 테스트(COCO128 서브셋 61장, 파이프라인 검증 목적) 결과:

| 조합 | 모델 | 라벨방식 | Precision | Recall | mAP50 |
|---|---|---|---|---|---|
| n_full | YOLO11n | 전체범위 | 0.907 | 0.748 | 0.812 |
| n_visible | YOLO11n | 가시영역만 | 0.649 | 0.297 | 0.304 |
| s_full | YOLO11s | 전체범위 | 0.961 | 0.891 | 0.953 |
| s_visible | YOLO11s | 가시영역만 | (미실행) | | |

**잠정 결론(6장에 반영됨)**: 전체범위 라벨링이 가시영역만 라벨링보다 확연히 우수. 모델 크기는 s가 n보다 정확도 우세하나 Jetson 실시간성과의 트레이드오프는 본 학습 후 재판단 필요.

**주의**: 위 수치는 61장의 매우 작은 표본 기준으로, 절대 성능 지표가 아니라 파이프라인 정상 동작 확인 및 상대적 경향 파악 용도. RTX 3090 환경에서 실제 데이터(공개 occlusion 서브셋 + 합성 + 자체촬영, 수만 장 규모)로 본 학습을 통해 재검증 필요.

## 9. 핵심 기능 요구사항

### FR-1. 잔해 매몰 인체 탐지 모델
- YOLO11 기반, 전체범위(full-extent) bbox 라벨링 방식 채택
- 성능 목표: occlusion 비율(30/50/70%)별 recall/precision을 정량 제시, 베이스라인(사전학습) 대비 개선 입증

### FR-2. 4방향 카메라 통합 처리
- Jetson에서 4채널 독립 추론, 각 채널=탐지 방향으로 매핑
- 실시간 처리 FPS 목표치 설정 및 RTX 3090/Jetson 각각 실측

### FR-3. GPS 위치 태깅
- 구조대원 위치 2초 주기 전송, 탐지 이벤트 발생 시점 GPS 좌표 매칭

### FR-4. 현장 알림 (부저 + LCD)
- 탐지 시 부저 알림(confidence≥0.5), LCD 알림(confidence≥0.25, 방향+미니맵 표시)

### FR-5. 서버 통신 및 저장
- 탐지 이벤트: {timestamp, gps_lat, gps_lon, 구조대원ID, 방향, confidence, 스냅샷}
- 네트워크 단절 대비 로컬 큐잉(SQLite)

### FR-6. 관제센터 대시보드
- 지도 위 구조대원 위치/경로, 탐지 마커(상태별 색상), 이력 리스트, 확인완료 처리

### FR-7. 구조대원 개인 뷰
- 개인 ID 로그인, 본인 탐지 이력 확인

## 11. 데모 및 검증 시나리오

### 11.1 데모 환경
- **시나리오 A (권장)**: 학교 운동장/공터에 잔해 모형(폐자재, 매트 등으로 부분 가림 구성) 설치, 실제 사람이 부분적으로 가려진 상태로 배치. 구조대원 역할자가 헬멧 유닛 착용 후 도보 이동하며 실시간 탐지 및 대시보드 반영 확인.
- **시나리오 B (보완)**: 산악 트레일 등 실제 야외 기복 지형에서 GPS 정확도 및 실사용성 검증.

### 11.2 검증 지표
- 탐지 recall/precision (occlusion 조건별)
- 탐지→서버→대시보드 반영 지연시간
- GPS 태깅 정확도(실측 위치와의 오차)
- 실사용자(구조대원 역할) 피드백: LCD 알림의 인지 용이성

## 14. 차별성 요약 (발표/보고서용)

1. 일반 사람 탐지가 아닌 **잔해 매몰/부분 가림 상황에 특화한 탐지 모델** — 기존 COCO 기반 범용 모델의 한계를 정량적으로 제시하고 개선. 초기 실험에서 라벨링 전략(전체범위 vs 가시영역)에 따른 성능 차이를 실증적으로 확인한 것 자체가 기술적 기여점.
2. SLAM 없이 GPS+탐지 좌표 태깅만으로 실용적인 현장 지도를 구성 — "완전 붕괴형 개활지"라는 현실적 시나리오 제약을 명확히 인지하고 설계.
3. 구조대원 개인 알림 + 관제센터 통합 관제라는 2단 구조로, 현장 대응과 지휘소 의사결정을 모두 지원.
